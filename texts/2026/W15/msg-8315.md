Ок. Я бы делал **именно как экспериментальный fallback path**, не как “основной продовый провайдер”.

### Что предлагаю изменить

**Цель:** поднять `claude-max-api-proxy` локально и направить OpenClaw в него для Opus.

### План
1. Проверить текущее состояние:
   - установлен ли `claude`
   - есть ли авторизация Claude Code CLI
   - где лежит текущий `openclaw.json`
   - какой сейчас активный model/base URL

2. Установить proxy:
   - `npm install -g claude-max-api-proxy`

3. Поднять proxy локально:
   - `claude-max-api`
   - проверить:
     - `http://localhost:3456/health`
     - `http://localhost:3456/v1/models`

4. Переключить OpenClaw на proxy
   - выставить `OPENAI_BASE_URL=http://localhost:3456/v1`
   - модель переключить на proxy-маршрут
   - аккуратно: это, скорее всего, будет что-то вроде `openai/claude-opus-4` или близкий идентификатор, в зависимости от того, что отдаст `/v1/models`

5. Проверить:
   - `openclaw models status`
   - `openclaw status`
   - тестовый запрос через OpenClaw на Opus

6. Если всё ок — опционально сделать автозапуск через LaunchAgent

---

### Важные риски
- Это **community workaround**, не официальный стабильный путь
- Может сломаться из-за изменений у Anthropic/Claude CLI
- Может потребовать ручной re-login в Claude CLI
- Часть нативных фич OpenClaw/Anthropic не будет работать как при прямом API-path

---

### Что именно будет затронуто
С побочными эффектами:
- глобальная npm-установка пакета
- изменение OpenClaw-конфига
- запуск локального фонового сервиса/proxy
- возможно LaunchAgent для автозапуска

---

### Моя рекомендация
**Сделать, но в 2 шага:**
1. сначала **ручной тест без автозапуска**
2. если стабильно работает — потом уже закрепить LaunchAgent

Это безопаснее и быстрее для отката.
