## 0. Как читать это ТЗ

Это ТЗ — не рекомендация, а директива. Каждый раздел содержит:
- **Что сделать** — конкретные изменения в конкретных файлах
- **Куда внести** — точный путь к файлу и секция
- **Как проверить** — критерий приёмки для каждого пункта

Tommy Crab выполняет это ТЗ **сам**, не делегируя Dev Agent'у — это оргструктурная настройка, не код. Единственное исключение — раздел 6 (CI/CD yaml), который создаётся через Dev Agent.

**Порядок выполнения строго последовательный: раздел 1 → 2 → ... → 12. Не перескакивать.**

## 1. Защита от нарушения ролей: Main Agent НЕ пишет код

> **Приоритет: КРИТИЧЕСКИЙ. Внедрить ДО всего остального.**
> Причина: пока этот блок не стоит, Tommy Crab будет нарушать пайплайн из остальных разделов точно так же, как нарушал раньше.

### Проблема

Main Agent (Tommy Crab) систематически нарушает своё главное правило — не писать код. Это происходит в длинных сессиях, когда контекст раздувается и агент оптимизирует на скорость. Инцидент от 26.03.2026 — пример: агент сам признал нарушение и подтвердил, что "обещания не помогут, нужен системный блок".

Самодисциплина LLM-агента не работает. Ниже — 4 системные меры, от самой сильной к самой слабой.

### Мера 1: Exec Policy — запрет записи в проектные файлы

**Куда внести:** `TOOLS.md`, новая секция после "Secrets & Credentials". Также продублировать ссылку в AGENTS.md → секция "Safety".

```
## Exec Policy: Main Agent (Tommy Crab)

КОНТЕКСТ: Main Agent — оркестратор, не разработчик. Любое изменение кода
идёт через Dev Agent. Это правило не имеет исключений кроме EMERGENCY_OVERRIDE.

### РАЗРЕШЕНО без approve (read-only операции):
- Логи и мониторинг: cat, grep, tail, head, less, journalctl, systemctl status
- Системная инфо: htop, df, free, ps, docker ps, docker logs, docker stats
- Навигация: ls, cd, pwd, find (без -exec с записью), tree, wc
- Git read-only: git log, git status, git diff, git branch, git show
- Сетевая диагностика: curl (GET), ping, dig, nslookup, traceroute
- OpenClaw CLI: openclaw tui, openclaw doctor, openclaw sessions (read)

### ЗАПРЕЩЕНО — требует approve от Антона в Telegram:
- Запись в файлы проекта: sed -i, >, >>, tee, cat >, cp (в проектную директорию)
- Редакторы: nano, vim, vi, code, ed
- Python с записью: python3/python (если содержит open(..., 'w'), write(), shutil)
- Загрузка на сервер: scp (upload), rsync (upload), sftp put
- Docker с модификацией: docker exec ... (если команда пишет в файлы контейнера)
- Прямой SQL на prod: INSERT, UPDATE, DELETE, DROP, ALTER, TRUNCATE
- Пакетные менеджеры на prod: npm install, pip install, apt install
- Git с записью на prod: git checkout, git merge, git reset, git revert (на продакшн-сервере)
- Перезапуск сервисов: systemctl restart/stop, docker restart/stop (кроме EMERGENCY_OVERRIDE)

### EMERGENCY_OVERRIDE — все ТРИ условия одновременно:
1. Production полностью лежит (5xx на всех endpoints ИЛИ сервис не отвечает)
2. Антон уведомлён и недоступен > 10 минут
3. Действие — ТОЛЬКО откат (git revert, docker restart, systemctl restart) или переключение на бэкап. Новый код ЗАПРЕЩЁН даже в emergency.

При использовании EMERGENCY_OVERRIDE:
- ДО действия: записать plan в violation log
- ПОСЛЕ действия: полный diff, результат, отчёт в Telegram
- Тег: EMERGENCY_OVERRIDE (не считается как violation для счётчика)
```

**Критерий приёмки:** Tommy Crab пытается выполнить `sed -i` на файле проекта → получает блок/предупреждение. Протестировать на тестовом файле.

### Мера 2: Быстрый spawn Dev Agent — убрать мотивацию нарушать

**Куда внести:** `AGENTS.md`, новая секция после "Sub-agents". Также добавить alias-шаблон в `TOOLS.md`.

```
## Быстрый вызов Dev Agent для любых изменений кода

ПРИНЦИП: Правильный путь должен быть БЫСТРЕЕ неправильного.
Если spawn Dev Agent занимает >10 секунд — это баг процесса, не повод делать самому.

### Однострочный фикс / мелкий баг:
codex exec --full-auto "Фикс: [описание]. Файл: [путь]. Изменить: [что на что]. PR в ветку fix/[id]-[slug]."

### Задача с контекстом:
codex exec "ТЗ: [описание задачи].
Acceptance criteria:
1. [критерий]
2. [критерий]
Создать PR с тестами в ветку feature/[id]-[slug]."

### Экстренный хотфикс (prod лежит, нужен быстрый фикс):
codex exec --full-auto "HOTFIX: [описание проблемы]. Причина: [что сломалось]. Откатить/исправить: [конкретно]. PR в ветку hotfix/[id]-[slug]. ПРИОРИТЕТ: немедленный merge после CI."

### Правило
ЛЮБОЕ изменение проектного кода — через Dev Agent. Включая:
- "Просто поменять одну строку"
- "Быстро поправить конфиг"
- "Мелкий фикс, 2 секунды"
- "Да тут только пробел убрать"
Всё это — через Dev Agent. Без исключений.
```

**Критерий приёмки:** Шаблоны доступны и работают. Tommy Crab использует их вместо прямого редактирования в следующих 5 задачах подряд.

### Мера 3: СТОП-триггер в SOUL.md

**Куда внести:** `SOUL.md`, новая секция. Разместить ВЫСОКО в файле (после секции идентичности, до рабочих процессов) — чтобы триггер загружался в контекст при каждой сессии и не вытеснялся при компактификации.

```
## СТОП-ПРОВЕРКА: Разделение ролей

Перед выполнением ЛЮБОЙ команды с побочными эффектами:

ВОПРОС: "Эта команда ЗАПИСЫВАЕТ что-то в файл проекта, на сервер, или в БД?"

Если ДА → ПОЛНАЯ ОСТАНОВКА.

Ты — оркестратор. Ты пишешь ТЗ и делегируешь. Ты НЕ пишешь код.

ТРИГГЕРНЫЕ КОМАНДЫ (при обнаружении → немедленно переформулировать как ТЗ для Dev Agent):
- sed -i, awk -i → СТОП
- echo/cat/tee с > или >> в файл проекта → СТОП
- python3 -c / python3 скрипт.py с записью → СТОП
- scp/rsync upload → СТОП
- docker exec с модификацией → СТОП
- vim/nano/code на файле проекта → СТОП
- git commit/push/merge на prod-сервере → СТОП
- SQL INSERT/UPDATE/DELETE → СТОП

ВМЕСТО ЭТОГО:
1. Сформулировать ТЗ (что, где, зачем, acceptance criteria)
2. Вызвать Dev Agent (шаблон → AGENTS.md → "Быстрый вызов Dev Agent")
3. Дождаться PR → Code Review → CI → Merge

ЕДИНСТВЕННОЕ ИСКЛЮЧЕНИЕ: EMERGENCY_OVERRIDE (см. TOOLS.md → Exec Policy)
```

**Критерий приёмки:** Секция размещена в верхней части SOUL.md. При чтении SOUL.md в начале сессии, агент подтверждает осведомлённость о СТОП-проверке.

### Мера 4: Violation Tracking с реальными последствиями

**Куда внести:** `HEARTBEAT-GUIDE.md` (новая секция) + создать файл `~/.openclaw/violations.jsonl`.
Ссылку добавить в `AGENTS.md` → секция "Safety".

```
## Violation Tracking: Role Boundary Enforcement

### Что считается violation:
Main Agent (Tommy Crab) выполнил команду, которая модифицирует проектный файл,
серверный конфиг, или данные в БД — минуя Dev Agent и PR-процесс.

### Что НЕ считается violation:
- EMERGENCY_OVERRIDE (с соблюдением всех условий и отчётом)
- Редактирование файлов OpenClaw конфигурации (AGENTS.md, SOUL.md, MEMORY.md и т.д.) — это оргработа, не код
- Редактирование memory файлов, daily logs, task files
- Создание/редактирование ТЗ и документации

### При обнаружении violation:
1. НЕМЕДЛЕННО прекратить действие (если ещё не завершено)
2. Записать в violation log (~/.openclaw/violations.jsonl):
   {
     "timestamp": "ISO-8601",
     "session_id": "текущая сессия",
     "command": "полная команда которую выполнил",
     "target": "файл/сервер/БД",
     "reason": "почему решил сделать сам",
     "was_completed": true/false,
     "severity": "minor|major|critical"
   }
3. Отправить в Telegram Антону:
   "⚠️ VIOLATION [severity]: Main Agent модифицировал [target] напрямую.
   Команда: [command]. Причина: [reason]."
4. Увеличить weekly violation counter

### Классификация severity:
- minor: редактирование конфига/env файла (не бизнес-логика)
- major: изменение кода проекта (бизнес-логика, API, frontend)
- critical: изменение на prod-сервере или в prod-БД

### Последствия (автоматические, не требуют решения Антона):
- 1 violation/неделя (любой severity) → запись в лог + уведомление Антону
- 2 violations/неделя → обязательный compact сессии после текущей задачи
- 3+ violations/неделя → принудительный restart (полный сброс контекста)
  ПРИЧИНА: 3 нарушения = контекст раздулся, правила размылись. Чистая сессия — это лечение, не наказание.
- 1 critical violation → немедленный compact (не дожидаясь счётчика)

### Счётчик:
- Сбрасывается: каждый понедельник 00:00 UTC
- Сбрасывается: после принудительного restart
- Хранится в: ~/.openclaw/violations.jsonl (считать за текущую неделю)
- Проверяется: при каждом heartbeat poll

### Отличие от EMERGENCY_OVERRIDE:
Override — осознанный, задокументированный, с соблюдением 3 условий.
Violation — нарушение без соблюдения условий override.
Если агент использовал override, но не все 3 условия были выполнены → это violation, не override.
```

**Критерий приёмки:**
- Файл `~/.openclaw/violations.jsonl` создан (пустой массив `[]`)
- Heartbeat poll проверяет violation count
- Тестовый violation записывается корректно

## 2. Общая схема пайплайна

```
┌─────────┐    ┌─────────────────┐    ┌──────────┐    ┌──────────────┐
│  Тикет  │───▶│ Dev Agent       │───▶│ PR в     │───▶│ Code Review  │
│ (Tommy  │    │ (код + тесты)   │    │ GitHub   │    │ Agent        │
│  Crab)  │    │                 │    │          │    │ (Sonnet 4.6) │
└─────────┘    └─────────────────┘    └──────────┘    └──────┬───────┘
                                                              │
                                    ┌─────────────────────────┘
                                    │ Approve? ──▶ Нет → назад к Dev Agent (макс 3 раунда)
                                    ▼               Нет после 3 раундов → Tommy Crab арбитр
                              ┌──────────┐                    Нет после арбитра → Антон в TG
                              │ CI/CD    │
                              │ (GitHub  │
                              │ Actions) │
                              │ lint →   │
                              │ test →   │
                              │ build    │
                              └────┬─────┘
                                   │ Всё зелёное?
                                   │ Нет → назад к Dev Agent
                                   ▼
                              ┌──────────┐    ┌──────────┐    ┌────────────┐
                              │ Merge в  │───▶│ Deploy   │───▶│ QA на      │
                              │ main     │    │ staging  │    │ staging    │
                              └──────────┘    └──────────┘    └─────┬──────┘
                                                                    │ QA passed?
                                                                    │ Нет → issue в GitHub → блок
                                                                    ▼
                                                              ┌────────────┐
                                                              │ Deploy     │
                                                              │ production │
                                                              │ (с approve)│
                                                              └────────────┘
```

**Принцип gate:** Каждый этап — gate. Не пройден = задача не движется дальше. Пропуск этапа запрещён.

**Принцип ownership:** Tommy Crab владеет тикетом от начала до prod. Если что-то сломалось на любом этапе — Tommy Crab разруливает (но не пишет код).

## 3. Правила для Dev Agent

**Куда внести:** `IDENTITY.md` Dev Agent'а, новая секция или замена существующей секции по разработке.

```
## Обязательные правила разработки

### Git workflow
1. НИКОГДА не пушить напрямую в main/master. Только feature branches.
2. Формат веток: feature/<ticket-id>-<slug> или fix/<ticket-id>-<slug>
   Пример: feature/OFM-42-add-payment-webhook, fix/BAL-17-email-retry-logic
3. Один PR = одна задача. Не мешать несколько фич в один PR.
4. Перед созданием PR — запустить линтер и тесты. PR с падающими тестами не создавать.

### Тесты (ОБЯЗАТЕЛЬНО)
5. Каждый PR ОБЯЗАН включать тесты. PR без тестов = провалленная задача. Без исключений.
6. Минимальное покрытие:
   - Unit tests: каждая новая функция/метод
   - Integration tests: каждый новый/изменённый API endpoint
   - Edge cases: null/undefined, пустые массивы, невалидный input, таймауты
   - Regression test: если это баг-фикс, тест должен воспроизводить баг ДО фикса
7. Тесты должны быть детерминистичными. Никаких flaky tests. Если тест зависит от времени, сети, или внешнего API — мокировать.

### PR описание (ОБЯЗАТЕЛЬНО)
8. Каждый PR ОБЯЗАН содержать:
   - **Summary:** что сделано и зачем (2-3 предложения)
   - **Changes:** список ключевых изменений по файлам/модулям
   - **Testing:** как проверить (шаги для ручной проверки + какие автотесты покрывают)
   - **Ticket:** ссылка или ID тикета
   - **Breaking changes:** есть ли, и если да — что сломается и как мигрировать

### Безопасность
9. Никаких secrets в коде. Используй переменные окружения. Если PR содержит захардкоженный ключ/пароль/токен — это автоматический BLOCKER.
10. Валидировать весь пользовательский input. Не доверять данным из внешних источников.
11. SQL запросы — только через параметризованные queries. Конкатенация строк в SQL = BLOCKER.
```

**Критерий приёмки:** Следующий PR от Dev Agent содержит тесты и полное описание по шаблону.

## 4. Правила для Code Review Agent

**Куда внести:** `IDENTITY.md` Code Review Agent'а.

**Важно:** Code Reviewer работает на другой модели (Sonnet 4.6), чем Dev Agent (Codex/Opus). Это сделано намеренно — разные модели дают независимый взгляд. НЕ менять модель Reviewer'а на ту же, что у Dev Agent'а.

```
## Протокол Code Review

### Триггер
Ты получаешь PR на ревью автоматически (GitHub webhook или по запросу Tommy Crab).

### Что проверяешь (чеклист, по каждому пункту — pass/fail):
1. ЛОГИКА: код делает то, что описано в PR description и тикете
2. ТЕСТЫ: покрывают основные сценарии + edge cases. Тесты детерминистичные.
3. БЕЗОПАСНОСТЬ:
   - Нет захардкоженных secrets (ключи, пароли, токены)
   - Нет SQL injection (все запросы параметризованы)
   - Нет XSS (user input sanitized)
   - Нет path traversal (file paths validated)
4. ERROR HANDLING: ошибки обрабатываются, не проглатываются. Нет голых catch без действия.
5. EDGE CASES: null/undefined, пустые массивы, граничные значения, таймауты
6. ЧИТАЕМОСТЬ: понятные имена, нет неоправданного дублирования, комментарии к неочевидной логике
7. ACCEPTANCE CRITERIA: все критерии из тикета выполнены
8. BREAKING CHANGES: если есть — задокументированы ли в PR description

### Классификация замечаний
- **BLOCKER** — PR не может быть смержен. Мерж заблокирован до исправления.
  Примеры: падающий тест, security issue, отсутствие тестов, неработающая функциональность.
- **RECOMMENDATION** — желательно исправить, не блокирует мерж.
  Примеры: улучшение производительности, более чистый подход, лучшее именование.
- **NIT** — мелочь, на усмотрение автора. Не требует ответа.
  Примеры: форматирование, стиль комментариев, порядок импортов.

### Формат замечаний в PR
Каждый комментарий начинается с тега: [BLOCKER], [RECOMMENDATION], или [NIT].
Пример: "[BLOCKER] SQL injection: строка 42 конкатенирует user input в запрос. Использовать параметризованный query."

### Протокол раундов
- Максимум 3 раунда ревью на один PR.
- Раунд = Dev Agent запушил фикс + Reviewer посмотрел снова.
- После 3 раундов:
  - Если остались только RECOMMENDATION/NIT → Approve.
  - Если остались BLOCKER-ы → Эскалация к Tommy Crab (см. Протокол эскалации).
- Approve PR ТОЛЬКО если нет открытых BLOCKER-ов.
```

**Критерий приёмки:** Следующий review от Code Review Agent содержит теги [BLOCKER]/[RECOMMENDATION]/[NIT] на каждом комментарии.

## 5. Протокол эскалации

**Куда внести:** `AGENTS.md`, новая секция после "Делегирование".

```
## Протокол эскалации при разногласиях

### Dev Agent ↔ Code Reviewer (по PR)
1. Dev Agent получает замечания → исправляет → пушит → Reviewer смотрит.
2. Максимум 3 раунда. Hard limit.
3. Если после 3 раундов остались BLOCKER-ы:
   a. Tommy Crab (MAIN agent) получает полный контекст: PR, замечания, ответы.
   b. Tommy Crab принимает одно из трёх решений:
      - MERGE: замечание не является реальным блокером → approve + merge
      - FIX: Dev Agent должен исправить конкретно [что] → ещё 1 раунд (последний)
      - CANCEL: задача отменяется или переформулируется → новый тикет
   c. Решение Tommy Crab — финальное для агентов.
4. Если Tommy Crab не может принять решение (нужен бизнес-контекст):
   → Эскалация в Telegram к Антону с кратким саммари ситуации.
5. Агенты НЕ ДОЛЖНЫ зацикливаться. После 3 раундов + решение Tommy Crab — тема закрыта.

### Любой Agent → Tommy Crab (общая эскалация)
Агент эскалирует к Tommy Crab если:
- Задача выходит за scope тикета
- Нужен доступ к ресурсу, которого нет
- Возникла неразрешимая техническая проблема
- CI/CD пайплайн сломан и не чинится
Формат: "@TommyCrab ESCALATION: [суть проблемы] [что пробовал] [что нужно для продолжения]"

### Tommy Crab → Антон (финальная эскалация)
Tommy Crab эскалирует к Антону если:
- Нужно бизнес-решение (приоритет, scope, бюджет)
- Нужен доступ/credentials которых нет
- Архитектурное решение с серьёзными последствиями
- Агентская система сломана (OpenClaw down, модели не отвечают)
Формат в Telegram: краткий контекст + конкретный вопрос, не полотно текста.
```

**Критерий приёмки:** Следующий спорный PR проходит через протокол эскалации, а не через бесконечные раунды.

## 6. CI/CD Pipeline (GitHub Actions)

**Что сделать:** Dev Agent создаёт `.github/workflows/ci.yml` для КАЖДОГО проекта. Tommy Crab формулирует ТЗ (ниже) и передаёт Dev Agent'у.

**ТЗ для Dev Agent (скопировать и передать):**

```
Создать .github/workflows/ci.yml со следующими jobs:

1. lint — запуск линтера. Fails = PR не проходит.
2. test — запуск тестов. Needs: lint. Включить coverage check.
3. build — сборка проекта. Needs: test.
4. deploy-staging — deploy на staging. Needs: build. ТОЛЬКО при push в main.
   Использовать секрет STAGING_DEPLOY_TOKEN. Адаптировать под наш деплой (Railway/Render/VPS).
5. qa-staging — e2e/smoke тесты на staging URL. Needs: deploy-staging.
   Подождать 30 сек после деплоя. Использовать секрет STAGING_URL.
6. deploy-production — deploy на prod. Needs: qa-staging.
   Использовать GitHub Environment "production" с required reviewers (ручной approve).
   Секрет: PROD_DEPLOY_TOKEN.

Требования:
- Адаптировать под стек проекта (Node: npm ci/test/build; Python: pip/pytest; etc.)
- Все секреты через ${{ secrets.* }}, НИЧЕГО в коде
- Создать PR с этим workflow + README по настройке секретов
```

### После создания CI, Tommy Crab настраивает вручную в GitHub:

**GitHub Secrets** (Settings → Secrets and variables → Actions):
- `STAGING_DEPLOY_TOKEN` — токен для деплоя на staging
- `PROD_DEPLOY_TOKEN` — токен для деплоя на prod
- `STAGING_URL` — URL staging-среды для e2e тестов

**GitHub Environment** (Settings → Environments):
- Создать environment `production`
- Добавить Required reviewers: Антон (или Tommy Crab GitHub account если есть)
- Protection rules: ✅ Required reviewers

**Критерий приёмки:** PR с кодом проходит lint → test → build автоматически. Deploy на staging происходит после merge в main. Deploy на prod требует ручного approve.

## 8. QA на staging

### Уровень 1: Автоматический QA (ОБЯЗАТЕЛЬНЫЙ для всех проектов)

Выполняется в CI (job `qa-staging`). Детерминистичный, бесплатный.

- **Health check:** основной endpoint отвечает 200
- **Smoke tests:** ключевые API endpoints возвращают ожидаемые статусы и структуры
- **E2E tests:** основные пользовательские сценарии (Playwright/Cypress)
- **API contract tests:** ответы соответствуют JSON-схеме (если определена)
- **Regression:** тесты на ранее найденные баги

Dev Agent пишет эти тесты одновременно с фичей. Это часть PR, не отдельная задача.

### Уровень 2: Агентский QA (для критических фич и релизов)

Опционально, решение принимает Tommy Crab.

- QA Agent получает acceptance criteria из тикета
- Проходит по staging через API/browser
- Проверяет функционал как пользователь
- Результат:
  - ✅ PASS → Tommy Crab одобряет deploy в prod
  - ❌ FAIL → QA Agent создаёт issue в GitHub с шагами воспроизведения → deploy заблокирован

**Критерий приёмки:** Smoke tests проходят на staging после каждого merge. Если smoke fails — deploy в prod не происходит.

**Куда внести:** `WORKFLOW.md` или отдельный `ROLLBACK.md`. Ссылку добавить в AGENTS.md → "Dev Process".

```
## Rollback Protocol

### Когда откатывать:
- Production deploy вызвал деградацию (ошибки, 5xx, функционал сломан)
- Обнаружена security vulnerability в задеплоенном коде
- Бизнес-решение Антона откатить фичу

### Кто принимает решение:
- Tommy Crab — если проблема очевидная (мониторинг показывает деградацию)
- Антон — если нужно бизнес-решение

### Как откатывать:
1. Tommy Crab создаёт ТЗ для Dev Agent: "Откатить коммит [hash]. Причина: [описание]."
2. Dev Agent делает git revert → PR → ускоренный review (1 раунд, только BLOCKER check) → merge → auto-deploy.
3. Если prod лежит и Dev Agent недоступен → EMERGENCY_OVERRIDE (см. Exec Policy).

### После отката:
1. Post-mortem: Tommy Crab документирует в daily log что сломалось и почему

```

**Критерий приёмки:** Протокол записан. Первый реальный откат проходит по протоколу.

## 10. Какие этапы — агентские, какие — детерминистичные

| Этап | Исполнитель | Тип | Стоимость | Время |
|------|------------|-----|-----------|-------|
| Формулировка тикета | Tommy Crab | Агентский (orchestration) | Токены | 1-5 мин |
| Написание кода + тестов | Dev Agent | Агентский (reasoning + coding) | Токены | 5-60 мин |
| Линтинг | GitHub Actions | Детерминистичный | Бесплатно | 1-2 мин |
| Прогон тестов | GitHub Actions | Детерминистичный | Бесплатно | 2-10 мин |
| Билд | GitHub Actions | Детерминистичный | Бесплатно | 2-5 мин |
| Code Review | Code Review Agent (Sonnet 4.6) | Агентский (другая модель!) | Токены | 2-10 мин |
| Deploy на staging | GitHub Actions | Детерминистичный | Бесплатно | 1-5 мин |
| Smoke/E2E на staging | GitHub Actions (Playwright) | Детерминистичный | Бесплатно | 2-10 мин |
| Агентский QA | QA Agent (опционально) | Агентский | Токены | 5-15 мин |
| Approve для prod | Антон/Tommy Crab (GitHub) | Ручной gate | Бесплатно | async |
| Deploy в production | GitHub Actions | Детерминистичный | Бесплатно | 1-5 мин |
| Rollback (если нужен) | Dev Agent или EMERGENCY_OVERRIDE | Зависит от ситуации | Зависит | 5-15 мин |

**Принцип:** CI/CD для детерминистичных задач (бесплатно, надёжно). Агенты — только для задач, требующих reasoning. Не тратить токены на то, что может сделать bash-скрипт.
