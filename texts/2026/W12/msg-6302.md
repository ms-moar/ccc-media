🔍 Анализ: OpenClaw Setup Prompt

Что это

Универсальный bootstrap-промпт для нового OpenClaw агента. По сути — "franchise kit" для клонирования продакшн-сетапа CEO-ассистента на новые инстансы. Хорошо структурированный, видно опыт продакшна.

✅ Что сильно

1. Позиционирование "Deputy CEO, не ассистент"
Правильный фрейминг. Агент-ассистент спрашивает "что делать?", а Deputy CEO думает "что нужно сделать?" и предлагает. Это принципиально меняет поведение модели.

2. "Ты НЕ пишешь код. Ни строки."
Самое важное правило. 90% провалов AI-агентов — orchestrator пишет код сам вместо делегации, тонет в деталях, теряет стратегический контекст. Жёсткий запрет = чистая архитектура.

3. Memory tagging система
⚠️ DECISION, ⚠️ LESSON, ⚠️ BLOCKER, 📝 LOG — элегантно. При сжатии decisions и lessons защищены от удаления. Это решает проблему "дефраг убил важное решение". У нас этого нет — стоит внедрить.

4. "Код vs Агент" framework для автоматизации
"Можно описать через if/else без интерпретации?" → bash. Нет → agent cron. Простое и правильное правило. Скрипт = $0. Агент = дорого. Мы иногда делаем agent crons где хватило бы скрипта.

5. Risk Policy (Low/Medium/High)
Явная классификация что можно автономно, что требует плана, что требует approval. У нас это размыто в AGENTS.md. Здесь — чётко.

6. Cross-business thinking
"Каждое решение в контексте портфеля" — для CEO с 10+ бизнесами это критично. Мы иногда оптимизируем один проект не видя whole picture.

⚠️ Что слабо / пропущено

1. MEMORY.md 200 строк — много
Автор пишет "лимит 200 строк" и "грузится в КАЖДЫЙ запрос → стоит денег" — но потом ставит 200. При Opus это ~4K токенов на каждый запрос только на MEMORY. Наш лимит 80 строк с lazy-load overflow — экономнее. Лучше бы 100 строк + указатели.

2. Нет HIPPOCAMPUS / sparse index
Memory структура (facts.md, decisions.md, projects/) хорошая, но нет индекса "где что искать". При 20+ проектах агент будет читать все файлы подряд. Hippocampus-подход решает это.

3. Нет post-session capture
Описан daily log и weekly compression, но нет механизма автоматической записи. Та же проблема что была у нас — зависит от "агент вспомнит записать". End-of-Day cron не упомянут.

4. Нет vault / canonical memory
Всё в workspace. Нет разделения на operational (черновики) и canonical (validated). Через полгода workspace превратится в помойку. Two-loop architecture (наш подход) надёжнее.

5. Deputy CEO per business — дорого без Synapse
"Для каждого бизнеса — отдельный Deputy CEO со своим workspace". Но как они делят контекст? Если Deputy CEO для Logity не знает что решил Deputy CEO для Fr8Mode — будут конфликты. Нужен Synapse protocol или хотя бы shared decisions.md.

6. Weekly compression вместо nightly defrag
Раз в неделю — слишком редко. За 7 дней накопится много, compression станет тяжёлой. Nightly defrag (наш подход, из whitepaper) лучше — инкрементальное сжатие проще и дешевле.

7. Нет decay / forgetting mechanism
"decisions.md — НИКОГДА не удаляются". Через год это 500+ решений. Нет приоритизации, нет decay. Файл станет шумом. Нужен хотя бы tiered access — recent decisions vs archived.

8. Нет baseline test / metrics
Как измерить что память работает? Нет метрик (coverage %, retrieval accuracy, boot cost). Без измерения — невозможно улучшать.

💡 Что стоит позаимствовать нам

1. Теги в daily logs (⚠️ DECISION, ⚠️ LESSON) — защита от deletion при сжатии. Внедрить в наш End-of-Day cron и defrag
2. Risk Policy (Low/Medium/High) — формализовать в нашем AGENTS.md вместо размытого "спроси если не уверен"
3. "Код vs Агент"  rule для cron jobs — проверять перед каждым новым кроном
4. facts.md как отдельный файл — у нас факты размазаны между MEMORY.md и context files. Dedicated facts store может быть полезен
5. /status и /goal команды — custom commands для quick situational awareness

Вердикт Solid 7/10. Продуманный bootstrap для нового агента. Видно реальный продакшн-опыт (теги, risk policy, "не пиши код").
